# 系统设计答题卡

这是一份适合放在旁边快速查看的简版提示卡，适用于：

- 系统设计面试
- 架构评审
- 轻量设计文档
- 后端服务方案讨论

核心目标不是“背组件”，而是让你的表达更有结构、更明确、更贴近工程实际。

## 1. 先把问题讲清楚

- 用户 / 调用方是谁？
- 核心请求或业务流程是什么？
- 本次讨论范围内包含什么？
- 明确不做什么？
- 最重要的目标是什么：延迟、吞吐、可靠性、正确性还是成本？

## 2. 先快速估规模

- DAU / MAU 大概多少？
- 峰值 QPS / RPS 多少？
- 读写比例如何？
- 未来 6–12 个月的数据增长怎样？
- 是否有热点 key、租户倾斜、突发流量？
- 延迟目标是什么（例如 p95 / p99）？

## 3. 先给高层架构图

第一版尽量简单：

`client -> API / gateway -> service -> cache / queue / database`

然后补充说明：

- 服务边界怎么划分
- 哪些部分是同步，哪些是异步
- 状态存在哪里
- 哪些链路在关键路径上

## 4. 说明 API 和契约

- 主要接口或 RPC 是什么
- 请求 / 响应结构
- 是否要求幂等
- 错误模型怎么定义
- 分页、过滤、重试怎么做

## 5. 说明数据模型

- 核心实体有哪些
- 主要访问模式是什么
- 为什么选这种存储
- 需要哪些索引
- 是否需要分区 / 分片
- Schema 演进风险在哪里

## 6. 讲清楚读路径

- 正常读请求怎么走
- 缓存放在哪里
- 缓存 miss 怎么办
- 热点 key 怎么处理
- 降级读策略是什么

## 7. 讲清楚写路径

- 校验、落库、异步任务的顺序
- 哪些步骤必须同步确认
- 是否要引入队列 / 事件流
- 重试策略如何
- 怎么做去重 / 幂等
- 部分成功时怎么补偿

## 8. 明确一致性和正确性

- 哪里必须强一致
- 哪里可以最终一致
- 哪些业务不变量绝不能破坏
- 可能出现哪些竞态
- 如何防止重复副作用

## 9. 讲故障和可靠性

- 依赖超时怎么办
- 重试 / 退避 / 熔断怎么放
- 队列积压或消费者故障怎么办
- 错误发布的影响面多大
- 有什么降级路径
- 故障后如何恢复

## 10. 补上可观测性和运维

- 哪些日志是排障必须的
- 关键指标有哪些
- 需要哪些 trace / span
- 第一版就应该有哪些告警和仪表盘
- 是否需要 runbook

## 11. 补上安全和滥用防护

- 认证 / 鉴权模型
- 敏感数据怎么处理
- 限流 / 风控怎么做
- 是否有审计要求
- 数据保留 / 删除要求是什么

## 12. 结尾一定讲 tradeoff 和 rollout

- 怎么安全发布
- 是否存在兼容性问题
- 回滚策略是什么
- 哪些内容推迟到 v2
- 最可能先失效的假设是什么

## 面试里很好用的开场句

- “我会先给一个满足当前规模和可靠性目标的简单方案。”
- “如果没有明确多地域要求，我会先做单 Region 版本。”
- “这个问题的核心 tradeoff 是正确性和延迟之间的平衡。”
- “我会先保证写路径幂等，再去优化性能。”
- “我想把故障模式和降级策略说清楚。”

## 常见收尾句

> 我会先上线一个简单、可观测、可回滚的版本，后续再根据真实流量和故障模式增加复杂度。
